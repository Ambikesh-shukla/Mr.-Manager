IMPORTANT GOAL:
This bot must be lightweight, scalable, cleanly coded, and optimized for smooth performance without sacrificing features. Avoid unnecessary files, duplicate handlers, bloated architecture, or overengineering. Keep code compressed, modular, and easy to maintain.

CRITICAL CHANGE:
Remove MongoDB and mongoose completely.
Do NOT remove any major features.
Keep the full advanced feature set, but replace database persistence with:
- JSON-based local storage
- fs/promises
- in-memory cache
- debounced save system
- safe file read/write helpers
- startup data loading
- automatic recovery for missing/corrupt files where possible

TECH STACK:

MAIN BOT PURPOSE:
A multi-purpose Discord bot focused on Minecraft server providers, hosting sellers, SMP owners, setup service sellers, and support teams.

1. ADVANCED TICKET SYSTEM (MAIN FEATURE)
Create a fully dynamic and customizable ticket system with all these features:

A. Ticket Panel Customization
Admins must be able to fully customize a ticket panel from Discord itself using slash commands / setup commands / modals:
- Panel title
- Panel description
- Embed color
- Footer
- Thumbnail
- Banner image
- Emoji
- Button label
- Dropdown label
- Dropdown placeholder
- Support category
- Allowed support roles
- Ping roles
- Ticket naming format
- Panel channel
- Ticket log channel
- Transcript log channel
- Ticket open message
- Ticket close message

B. Ticket Creation Types
Support both:
- Button-based ticket creation
- Dropdown/select menu based ticket creation

Admin must be able to choose:
- Single button panel
- Multi-button panel
- Dropdown with multiple ticket types

Examples of ticket types:
- Purchase
- Support
- Bug Report
- Partnership
- Staff Application
- Custom type defined by admin

Each ticket type should support:
- Custom emoji
- Custom label
- Custom description
- Custom category
- Custom support role

C. Ticket Modal System
Admin can enable optional modal forms before ticket creation.
Example fields:
- Minecraft username
- Problem summary
- Hosting plan interested in
- Budget
- Server version
- Extra details

Admin should be able to customize modal fields per ticket type.

D. Ticket Permissions
When a ticket is created:
- The user who opened the ticket can view and use it
- Admins can view and manage it
- Permitted support roles can view and manage it
- Optional extra permitted members can be added
- The ticket creator must have the right to close their own ticket
- Admins can close tickets
- Permitted support team members can close tickets

E. Ticket Controls Inside Ticket
Inside every ticket channel, send a control panel with buttons like:
- Close Ticket
- Claim Ticket
- Unclaim Ticket
- Add User
- Remove User
- Rename Ticket
- Change Ticket Type
- Save Transcript
- Reopen Ticket (if enabled)
- Delete Ticket

Permission logic:
- Ticket owner can close ticket
- Admin/support can close ticket
- Only support/admin can claim/unclaim
- Only support/admin can add/remove users unless config allows otherwise

F. Claim System
- Staff can claim a ticket
- Show claimed by whom
- Prevent multiple claims if configured
- Allow unclaim
- Log claim/unclaim actions

G. Close System
When closing ticket:
- Ask confirmation
- Generate transcript
- Send transcript to logs
- Save metadata in local JSON storage
- Optionally DM transcript summary to ticket creator
- Delay before deletion configurable
- Support close reason
- Support closed by tracking
- Save timestamps

H. Ticket Cooldown + Delay System
This is mandatory:
- Admin can set cooldown per panel or ticket type
- If user opens a ticket, they cannot create another ticket of the same configured type if blocked by rules
- Admin can choose:
  - Only 1 open ticket at a time
  - Multiple open tickets allowed
  - Cooldown after closing a ticket
- Example:
  user closes one ticket -> must wait X hours before opening another
- Cooldown must be stored in local JSON files and survive bot restarts
- Show friendly message telling remaining cooldown time

I. Ticket Limits
Support settings like:
- Max open tickets per user
- Max open tickets globally per panel
- Restrict duplicate ticket types
- Optional blacklist system for ticket creation
- Optional account age requirement
- Optional server join age requirement

J. Ticket Transcripts
Generate clean HTML transcript or clean formatted text transcript.
Store:
- Ticket creator
- Staff involved
- Open time
- Close time
- Claimed by
- Ticket type
- Reason
- Full transcript metadata
Send transcript to configured log channel.

K. Ticket Logs
Log all major actions:
- Ticket created
- Ticket closed
- Ticket deleted
- Ticket claimed
- Ticket unclaimed
- User added
- User removed
- Ticket renamed
- Panel updated
- Cooldown blocked
- Permission denied attempts

L. Ticket Config Management
Admins need slash commands to:
- Create ticket panel
- Edit panel
- Delete panel
- Add ticket type
- Edit ticket type
- Remove ticket type
- Set support roles
- Set panel channel
- Set log channel
- Set transcripts channel
- Set cooldown
- Set max tickets
- Enable or disable modal
- Enable or disable claim system
- Enable or disable transcript system
- Enable or disable reopen system
- Reset ticket config

Replace MongoDB schemas with file-based storage collections or managers for:
- Guild settings
- Ticket panels
- Ticket types
- Ticket data
- User cooldowns
- Claim state
- Transcript metadata
